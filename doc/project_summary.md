# Uzbek Text Simplifier — сводка проекта

Капстоун-проект: сервис, упрощающий сложные узбекские тексты (юридические
документы, официальные новости, гос. инструкции) для обычного читателя,
с опциональным переводом результата на русский/английский.

## Идея

Выбор пал на **NLP для узбекского языка** — нишу почти без готовых
инструментов (в отличие от английского). Из вариантов (классификация
новостей / тональность / NER / упрощение текста) выбрано **text
simplification** — сильнее по бизнес-ценности и без готовых конкурентов
под узбекский.

**Архитектура:**
1. Сбор parallel-корпуса (сложный ↔ простой текст)
2. Fine-tuning mT5-small на корпусе
3. NLLB-200 — опциональный перевод упрощённого текста на ru/en
4. Демо на FastAPI + Streamlit/Gradio, деплой на HuggingFace Spaces

## Сбор корпуса

Основные источники собраны серией парсеров с единой CSV-схемой (source,
title, url, chunk_id, text, word_count, length, avg_sentence_len,
avg_word_len, rare_word_ratio, complexity_score, domain). Чанкинг —
100–200 слов по границам предложений.

- Новости: kun.uz, gazeta.uz, uza.uz (RSS)
- Wikipedia: 14 категорий, рекурсивный обход подкатегорий (до 2 уровней)
- lex.uz: юридический корпус, 9 документов (ручной seed-список) — самый
  "сложный" стилистический полюс

Помимо основной серии (`uz_news_parser*.py`, `uz_yur_parser.py`) в
проекте есть ещё несколько парсеров сбора данных. Статус двух из них
прояснился: `uz_wiki_parser_1000.py` и `uz_wiki_parser_new.py` — это
рабочие прогоны, их результат уже вошёл в объединённый корпус (см.
ниже). Статус `uz_sports_parser.py` (спортивный корпус) пока отдельный
— он не участвует в `uz_corpus_merged_final_fixed.csv`, лежит в своём
файле (`uz_corpus_sports_*.csv`) и имеет собственную схему полей
(включает `subcategory`, которого нет в основном корпусе).

Сложность текста считается тремя эвристиками (без готовых NLP-инструментов
под узбекский): средняя длина предложения, доля редких слов, средняя длина
слова → взвешенный `complexity_score`.

## Объединение и очистка

Изначально было собрано ещё три дополнительных прогона парсера поверх
основного датасета (2033 строки). Позже к объединению добавились ещё
два прогона Wikipedia-парсера (`uz_wiki_parser_1000.py`,
`uz_wiki_parser_new.py`). Объединение и дедуп (по тексту чанка, а не по
url+chunk_id, т.к. один текст мог получать разный chunk_id в разных
прогонах) дали:

- **Итог: 2572 строки**, ~297 000 слов
- `domain` отсутствовал в новых файлах-прогонах — восстановлен
  эвристически по ключевым словам для 197 строк (помечены
  `domain_filled_auto`, рекомендуется выборочная ручная проверка,
  особенно в `other`/`technology`/`politics`, где выше пересечение
  лексики)
- Источники: wikipedia 1149, lex.uz 1071, gazeta.uz 124, uza.uz 124,
  kun.uz 104
- Домены: law 1279 (~50%), science 289, economy 182, culture 175,
  sport 159, politics 117, other 99, technology 96, health 96,
  education 80
- Перекос в `law` (за счёт lex.uz) стал заметно мягче после добавления
  Wikipedia-прогонов (было ~59%, стало ~50%), но домены
  health/technology/education всё ещё самые малочисленные (<100 строк
  каждый) — для fine-tuning по-прежнему стоит либо даунсэмплить law,
  либо догонять эти три домена

> Промежуточный экспортный файл объединённого корпуса какое-то время
> был повреждён (каждая строка обёрнута в лишний слой CSV-кавычек +
> мусорный хвост) — это исправлено, актуальный файл и цифры выше
> посчитаны из починенной версии. Подробности — в `docs/merge_report.md`.

Отдельно, ещё на этапе сбора 2033 строк, была очистка мусора разметки
(LaTeX-дампы, HTML-теги, обрывки твит-виджетов, zero-width символы):
изменено 45 строк из 2033, пустых строк после чистки не возникло. Эта
чистка предшествовала обоим этапам слияния и отдельно не перепроверялась
на новых строках — см. заметку в `docs/cleaning_report.md`.

## Инструкции для teacher-модели (генерация parallel-пар)

Найдено две версии инструкций для упрощения текста, задающие **разные
стратегии**:
- **v1** — консервативная: сохранить 85–95% исходной информации, не
  сокращать по объёму
- **v2** — агрессивная: сократить длину на 40–60%, разрешено убирать
  даты, номера статей закона, статистику ради простоты

Это не разные формулировки одного и того же, а разный подход к задаче.
Нужно явно выбрать одну версию перед генерацией parallel-пар — иначе
датасет получится методологически неоднородным (часть пар "почти
дословный пересказ", часть — "сжатый пересказ").

## Проверка качества

Прогнан через DataStory (Tkinter + Gemini) и ручной аудит. Автоматические
"тревоги" AI-анализа (уникальность `text`, "выбросы" в длине чанков,
будущие даты 2026) — ложные срабатывания, ожидаемые для NLP-датасета.
Ручной аудит дополнительно выявил утечку разметки (LaTeX/HTML/твиттер),
не пойманную парсером и DataStory — исправлено. Отдельно найдена и
исправлена порча CSV-экспорта объединённого корпуса (см. выше и
`docs/merge_report.md`) — после починки: 0 пустых `text`, 0 дублей по
тексту, 0 пропущенных значений в остальных колонках.
